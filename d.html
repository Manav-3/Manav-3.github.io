DBMS Overview

A Database Management System is software that enables users and applications to define, create, maintain, secure, and query a collection of related data. It abstracts physical storage details and offers controlled, consistent, and concurrent access to data. Typical examples include MySQL, PostgreSQL, Oracle Database, and SQL Server.

What a DBMS provides

  • Data definition and schema management
  • Data manipulation via query languages
  • Transaction processing with ACID properties
  • Security, authorization, and auditing
  • Backup, recovery, and concurrency control

Why it matters

  • Reduces redundancy through normalization
  • Ensures integrity via constraints
  • Improves performance with indexes and optimization
  • Supports sharing and collaboration safely

Data vs Information, Features of Information

Data

Raw facts, figures, or observations without context. Examples: 42, "2025‑09‑04", "Kathmandu".

Information

Processed, organized, or structured data that is meaningful and useful for decision‑making. Example: "Average temperature in Kathmandu on 2025‑09‑04 was 24°C".

Features of Good Information

  • Accuracy and reliability
  • Timeliness and relevance
  • Completeness and consistency
  • Conciseness and clarity
  • Cost‑effectiveness

Database and its Purpose

A database is an organized collection of interrelated data stored to model some domain. Its purpose is to support efficient retrieval, insertion, update, and deletion while preserving integrity and security. It enables shared access, eliminates anomalies, and scales with growth.

Key Terminologies

Table, Field, Record, Tuple, Object, Keys

  • Table (Relation): Set of rows and columns representing an entity type.
  • Field (Column/Attribute): Named property of the entity; each column has a data type and constraints.
  • Record (Row): One instance of an entity; a collection of field values.
  • Tuple: Theoretical term for a row in relational algebra.
  • Object: In object databases, encapsulates state (attributes) and behavior (methods).
  • Keys: Attributes that identify and relate records. See below.

Common Key Types

  • Candidate Key Minimal attribute set that uniquely identifies a record.
  • Primary Key Chosen candidate key used as the main identifier; not null and unique.
  • Alternate Key Candidate key not selected as primary.
  • Foreign Key Attribute in one table referencing a candidate/primary key in another table.
  • Composite Key Two or more attributes together uniquely identify a row.
  • Surrogate Key Artificial key (e.g., auto‑increment id) used when natural key is complex.

Data Types

Numeric

  • Integer types
  • Floating‑point
  • Fixed‑precision decimal

Character & String

  • CHAR (fixed length)
  • VARCHAR (variable length)
  • TEXT/CLOB

Date/Time & Others

  • DATE, TIME, TIMESTAMP
  • BOOLEAN
  • BLOB/BINARY, JSON, ENUM

Data Dictionary

A data dictionary is a centralized repository of metadata: table names, columns, data types, constraints, indexes, views, procedures, and access rights. It helps developers, analysts, and DBAs understand structure, lineage, and rules. In many DBMSs, it is exposed as system catalogs.

DBMS: Introduction, Objectives, Advantages, Disadvantages

Introduction & Objectives

  • Provide logical data independence from physical storage
  • Ensure integrity, consistency, and concurrent access
  • Enable expressive querying and reporting
  • Support security, auditing, backup, and recovery

Advantages & Disadvantages

  • Advantages: reduced redundancy, better integrity, shared data, scalability, performance tuning
  • Disadvantages: complexity, cost, single point of failure (if centralized), need for skilled administration

Types of Database Model

Hierarchical Model

Data is organized as a tree of parent–child records. One parent can have many children; each child has exactly one parent. Efficient for one‑to‑many navigation but inflexible for many‑to‑many.

Network Model

Generalizes the hierarchical model using sets and pointers, allowing a record to have multiple parents. Flexible but complex and tightly coupled to physical paths.

Relational Model

Data is represented as relations (tables) manipulated by relational algebra and SQL. Emphasizes logical independence, declarative queries, and normalization.

Entity–Relationship (ER) Model

Conceptual model describing entities, attributes, and relationships with cardinalities. Often used at design time and later mapped to relational schemas.

Integrity Constraints and Types

Domain Integrity

Each attribute value must be from its declared domain and satisfy checks.

age INT CHECK (age BETWEEN 0 AND 150)

Entity Integrity

Primary keys are unique and not null, guaranteeing that each row is identifiable.

PRIMARY KEY (student_id)

Referential Integrity

Foreign keys must reference existing primary/candidate keys or be null, maintaining consistency across tables.

FOREIGN KEY (dept_id) REFERENCES department(id)

Normalization

Introduction

Normalization is the stepwise decomposition and restructuring of tables to remove redundancy and update anomalies while preserving dependencies.

Normal Forms

  • 1NF: No repeating groups; atomic values per cell.
  • 2NF: In 1NF and every non‑key attribute fully depends on the whole candidate key (no partial dependency on part of a composite key).
  • 3NF: In 2NF and no transitive dependency of non‑key attributes on keys.

Advantages

  • Eliminates redundancy and anomalies
  • Improves integrity and clarity of schema

Disadvantages

  • More tables and joins may impact query performance
  • Over‑normalization can complicate reporting
BCNF, 4NF, and 5NF further refine dependency rules for special cases like overlapping keys and multi‑valued dependencies.

Centralized and Distributed Database

Introduction

Centralized databases keep all data at a single site. Distributed databases split or replicate data across multiple sites connected by a network, coordinated by a distributed DBMS.

Diagram

Advantages

  • Centralized: simpler management, strong consistency, easier backup
  • Distributed: locality of access, fault tolerance, scalability, availability

Disadvantages

  • Centralized: single point of failure, network latency for remote users
  • Distributed: complex design, distributed transactions, replication lag

Database Security

Introduction

Database security protects confidentiality, integrity, and availability of data against threats such as unauthorized access, injection, privilege misuse, and ransomware.

Challenges

  • Insider threats and excessive privileges
  • Weak authentication and poor encryption
  • SQL injection and misconfigured applications
  • Backup/replication exposure

Security Measures

  • Least privilege, role‑based access control
  • Strong authentication and MFA
  • At‑rest and in‑transit encryption
  • Input validation and prepared statements
  • Auditing, monitoring, regular patching, backups

DBA and Roles of DBA

The Database Administrator installs, configures, secures, monitors, and tunes databases. Roles include capacity planning, schema design review, backup and recovery strategy, performance tuning, user management, patching, and compliance reporting.

ER Diagrams

Introduction

Entity–Relationship diagrams visualize entities, attributes, and relationships with cardinalities and degrees to build a clear conceptual design.

Core Concepts

  • Entity: Real‑world object or concept represented as a rectangle.
  • Attribute: Property of an entity, drawn as an oval; key attributes are underlined.
  • Relationship: Association between entities, drawn as a diamond; annotated with cardinality (1:1, 1:N, M:N).
  • Cardinality & Degree: Number of instances related and number of participating entity types.

Symbols: rectangle (entity), oval (attribute), underlined key attribute, diamond (relationship), annotated cardinalities.

  dept_id INT,

  CONSTRAINT fk_dept FOREIGN KEY (dept_id) REFERENCES department(id)

UPDATE student SET dept_id = 1 WHERE roll_no = 101;
DELETE FROM student WHERE roll_no = 999;
SELECT name, email FROM student WHERE dept_id = 1 ORDER BY name;

Sequential vs Random Files

  • Random: direct access to arbitrary positions using offsets

Steps to Work with Files

  1. Define file pointer: FILE *fp;
  2. Open with required mode: fp = fopen("data.txt","r");
  3. Perform read/write/append operations
  4. Close the file: fclose(fp);

while(fgets(line,128,fp)) { /* process line */ }
fclose(fp);

Appending

fputs("more\n", fp); fclose(fp);
